WORK PACKAGE 06: Content rights and AI consent

Goal:
- Give publishers and individual authors machine-readable control over AI training consent and content rights, expressed in HTML meta tags, HTTP headers, feed metadata, and `ai.txt`.

Primary targets:
- `inc/rights.php`
- `tests/phpunit/test-rights.php`

Depends on:
- WP-01 (adapter interface for per-author consent resolution).

Problem statement:
- Publishers and writers increasingly need to express preferences about AI training use of their content. Current mechanisms (`robots.txt`, `ai.txt`, TDM headers) are site-wide. No WordPress plugin offers per-author or per-post AI consent that flows into structured metadata across feeds and HTML. On multi-author sites, different authors may have different preferences — a staff reporter may want maximum reach while a guest contributor opts out.

Planned changes:

1. Register per-author AI consent user meta:
   ```php
   register_meta( 'user', 'byline_feed_ai_consent', [
       'type'         => 'string',
       'single'       => true,
       'show_in_rest' => true,
       'default'      => '',
   ] );
   ```
   Values: `'allow'`, `'deny'`, `''` (unset, inherits site default).

2. Register per-post AI consent override:
   ```php
   register_post_meta( '', '_byline_ai_consent', [
       'type'         => 'string',
       'single'       => true,
       'show_in_rest' => true,
       'default'      => '',
   ] );
   ```
   Values: `'allow'`, `'deny'`, `''` (unset, inherits from author).

3. Consent resolution logic:
   ```php
   function byline_feed_resolve_ai_consent( \WP_Post $post ) : string {
       // Post-level override takes precedence
       $post_consent = get_post_meta( $post->ID, '_byline_ai_consent', true );
       if ( in_array( $post_consent, [ 'allow', 'deny' ], true ) ) {
           $consent = $post_consent;
       } else {
           // Fall back to most restrictive author preference
           $authors = byline_feed_get_authors( $post );
           $consents = array_filter( array_column( $authors, 'ai_consent' ) );
           if ( in_array( 'deny', $consents, true ) ) {
               $consent = 'deny';
           } elseif ( in_array( 'allow', $consents, true ) ) {
               $consent = 'allow';
           } else {
               $consent = '';
           }
       }
       return apply_filters( 'byline_feed_ai_consent', $consent, $post );
   }
   ```
   When multiple authors have conflicting preferences, the most restrictive preference wins (deny beats allow). This is the safe default — an author's opt-out should not be overridden by a co-author's opt-in.

4. HTML meta tag output (`wp_head` hook, optional strict mode):
   - If resolved consent is `deny`, optionally emit restrictive snippet directives:
     ```html
     <meta name="robots" content="nosnippet, max-image-preview:none" />
     ```
   - These directives affect search result presentation, not model-training guarantees.

5. HTTP header output:
   - Hook `send_headers` or `wp_headers` for singular posts.
   - If resolved consent is `deny`, add TDM reservation headers:
     ```
     tdm-reservation: 1
     tdm-policy: https://example.com/tdm-policy
     ```
   - Optionally emit a restrictive `X-Robots-Tag` in strict mode:
     ```
     X-Robots-Tag: nosnippet
     ```
   - The policy URL can be configured in plugin settings or default to a generated page.

6. Feed-level rights metadata:
   - `cc:license` is for explicit content license declarations only. Never use it for AI opt-out or policy URLs.
   - In RSS2 item output (extending WP-02), output `<cc:license>` only when a real license URL is configured (for example a Creative Commons URL).
   - For deny-policy signaling in feeds, use a dedicated extension element (for example `<byline:rights-policy href="https://example.com/tdm-policy" mode="deny"/>`) only when extension semantics are defined; otherwise keep deny-policy signaling in HTTP headers/meta.

7. Crawler policy generation (`ai.txt` + optional global `robots.txt` token support):
   - Register a rewrite rule for `/ai.txt` as an informational policy endpoint.
   - `robots.txt` directives are site-wide/path-wide controls. Treat them as a global mode, not a per-author/per-post enforcement mechanism.
   - Add optional `robots_txt` filter output for known AI-related crawler tokens, for example:
     ```
     User-agent: Google-Extended
     Disallow: /
     ```
   - Keep token lists configurable because crawler ecosystems change quickly.
   - Settings page with a simple textarea override for full manual control.

8. User profile field:
   - Add "AI Training Consent" field to user profile edit screen.
   - Radio buttons: "Allow", "Deny", "No preference (inherit site default)".
   - Description: "Controls whether your content may be used for AI model training. When set to Deny, machine-readable signals are added to pages and feeds containing your work."

9. Post editor control:
   - Add to the block editor sidebar (could share the `PluginDocumentSettingPanel` with the perspective control from WP-03).
   - Select: "Inherit from authors", "Allow AI training", "Deny AI training".

10. Write PHPUnit tests (`tests/phpunit/test-rights.php`):
    - Single author, consent=deny: policy headers present (`tdm-reservation` + `tdm-policy`), optional strict snippet directives present when enabled.
    - Single author, consent=allow: deny-policy signals absent.
    - Single author, consent unset: no output (inherits nothing).
    - Multiple authors, one deny + one allow: resolved consent is deny.
    - Post-level override beats author-level preference.
    - Filter overrides everything.
    - `ai.txt` endpoint returns expected policy content.
    - `robots_txt` output contains configured site-wide crawler-token rules when global mode is enabled.
    - Feed items for denied posts include rights metadata.

Validation:
- User profile shows AI consent field.
- Post editor shows AI consent override.
- Singular pages for denied posts emit configured policy signals (`tdm-reservation`, `tdm-policy`, and optional strict snippet directives).
- HTTP response headers include `tdm-reservation` and `tdm-policy` for denied posts.
- Feed output includes rights metadata for denied posts.
- `/ai.txt` returns valid content.
- `robots.txt` output includes configured site-wide AI-crawler token directives when global mode is enabled.
- Multi-author conflict resolution correctly applies most-restrictive-wins.
- PHPUnit suite green.
- PHPCS clean.

Notes:
- There is no universal, enforceable "do not train" directive. Signals are advisory and crawler-dependent. Documentation must clearly describe what each signal does and does not do.
- `Google-Extended` is currently one of the clearest crawler controls for whether Google may use content to improve Gemini Apps and Vertex AI generative APIs (and for grounding in Google Search). It still does not enforce compliance outside participating crawlers.
- The TDM Reservation Protocol (W3C draft) is still evolving. Implementation should track the spec and update header behavior as it stabilizes.
- Per-author consent in structured metadata is genuinely novel. Nobody is doing this. It will attract attention from the journalism and rights management communities. The documentation and communication around this feature should be precise about what it does and does not guarantee.
- This work package can be developed in parallel with WP-02 through WP-05 after WP-01 is complete, but the feed integration (step 6) depends on WP-02 being in place.
